# Chip Factor Autoresearch Agent Design

## Status

Draft created on 2026-04-29. This is a design document only.

This document adapts the autonomous research-loop idea from `../auto-research-karpathy` to this project's stock chip-factor research workflow. The goal is not autonomous trading. The goal is an auditable AI research agent that proposes, tests, analyzes, and records chip-analysis factors.

## Core Idea

The system should behave like an automatic research assistant for chip-analysis factors:

1. Work inside a user-specified stock universe.
2. Work inside a user-specified historical date range.
3. For each sample, collect the first `N` trading days after the sample start date.
4. Generate chip-change features over those `N` days.
5. Generate or apply candidate strategy rules.
6. Observe forward returns at `N+1`, `N+3`, and `N+5`.
7. Record every API call, feature calculation, agent judgment, backtest result, and final conclusion to disk.
8. Promote, observe, discard, or invalidate candidate rules based on evidence.

## Design Inspiration

The `auto-research-karpathy` example is useful because it is strict about research boundaries:

- one fixed evaluation metric,
- one fixed experiment loop,
- clear editable and non-editable surfaces,
- baseline first,
- keep/discard/crash decisions,
- durable experiment logs.

For this project, the equivalent structure is:

| Autoresearch concept | Chip-factor equivalent |
| --- | --- |
| `program.md` | `chip-factor-research-program.md` or this design document |
| `train.py` editable surface | candidate factor specs and candidate strategy rules |
| fixed `val_bpb` metric | fixed `N+1`, `N+3`, `N+5` backtest metrics |
| `results.tsv` | auditable research run result table |
| keep/discard/crash | promote/observe/discard/invalid |
| fixed 5-minute training budget | fixed stock universe, date range, sample window, and observation windows |

## Observation Windows

Target forward observation windows:

- `N+1`
- `N+3`
- `N+5`

Reasoning:

- `N+1` captures immediate short-term confirmation or failure.
- `N+3` captures short swing behavior without drifting too far from the signal.
- `N+5` captures roughly one trading week after the signal.

Current implementation note:

- The current app implementation was previously changed to `N+3`, `N+7`, and `N+15`.
- The next implementation phase should migrate the backend, frontend, tests, and docs to `N+1`, `N+3`, and `N+5`.

## Research Scope Inputs

Every research run must declare scope before execution:

```yaml
run_id: run-001
stock_universe:
  mode: manual
  symbols:
    - 000001.SZ
    - 600519.SH
date_range:
  start_date: "20240101"
  end_date: "20241231"
sample_window_days: 10
observation_offsets: [1, 3, 5]
strategy_candidates:
  - retained_accumulation_buy_watch
  - overhead_digestion_hold_to_buy_watch
```

Stock universe modes:

- `manual`: user enters explicit stock codes.
- `file`: user provides a local stock list file.
- `index_constituents`: future mode, if index membership data is added.
- `screened`: future mode, generated by market-cap, liquidity, or data-quality filters.

First implementation should use `manual` only.

Date range behavior:

- User provides `start_date` and `end_date`.
- The system creates sample starts from valid trading days within the range.
- For each sample start, the backend resolves `N + max(observation_offsets)` forward trading days.
- Samples without enough future trading days are marked `invalid`, not silently skipped.

## Mandatory Audit Logging

Every research run must create a durable directory:

```text
docs/research-runs/<run_id>/
  run-config.yaml
  api-calls.jsonl
  feature-snapshots.jsonl
  agent-decisions.jsonl
  backtest-results.tsv
  failure-cases.jsonl
  final-report.md
```

These files are append-only during a run.

### `run-config.yaml`

Purpose:

- freezes the experiment scope and prevents later ambiguity.

Required fields:

```yaml
run_id: string
created_at: ISO-8601 timestamp
stock_universe: object
date_range: object
sample_window_days: integer
observation_offsets: [1, 3, 5]
candidate_rules: list
baseline_strategy: string
git_commit: string
notes: string
```

### `api-calls.jsonl`

Purpose:

- records every backend/Tushare-facing data request made during the run.

Each line:

```json
{
  "timestamp": "2026-04-29T10:00:00Z",
  "run_id": "run-001",
  "request_id": "uuid",
  "source": "tushare",
  "endpoint": "cyq_chips",
  "ts_code": "000001.SZ",
  "params": {
    "trade_date": "20240102"
  },
  "status": "OK",
  "row_count": 106,
  "duration_ms": 312,
  "error_code": null,
  "error_message": null
}
```

Rules:

- Never log tokens or credentials.
- Log parameters enough to reproduce the data request.
- If a request fails, record error code and message.
- If data is cached, record `source: cache` and the cache key.

### `feature-snapshots.jsonl`

Purpose:

- records the feature state used by rules.

Each line:

```json
{
  "timestamp": "2026-04-29T10:01:00Z",
  "run_id": "run-001",
  "sample_id": "000001.SZ-20240102-N10",
  "ts_code": "000001.SZ",
  "analysis_start": "20240102",
  "signal_date": "20240115",
  "features": {
    "profit_ratio_delta": 6.2,
    "loss_ratio_delta": -5.8,
    "concentration_width_70_delta": -0.013,
    "retained_low_chip_ratio": 0.82
  },
  "quality": {
    "status": "OK",
    "chip_days": 10,
    "price_days": 10
  }
}
```

Rules:

- Feature values must be calculated before agent interpretation.
- Missing values must be explicit `null`.
- Quality status must be recorded with every feature snapshot.

### `agent-decisions.jsonl`

Purpose:

- records every AI-agent analysis, judgment, and rule decision.

Each line:

```json
{
  "timestamp": "2026-04-29T10:02:00Z",
  "run_id": "run-001",
  "agent": "hypothesis-agent",
  "sample_id": "000001.SZ-20240102-N10",
  "input_refs": [
    "feature-snapshots.jsonl:42",
    "docs/references/chip-change-strategy-traceability.md"
  ],
  "decision_type": "rule_signal",
  "candidate_rule": "overhead_digestion_hold_to_buy_watch",
  "action": "HOLD",
  "confidence": 0.62,
  "reasoning_summary": "Loss ratio decreased while price return was non-negative; concentration improved but retained low chips were not strong enough for BUY.",
  "evidence_level": "FACTOR_FAMILY",
  "status": "pending_backtest"
}
```

Rules:

- Reasoning must be summarized, not hidden.
- Every decision must point to input references.
- Every rule signal should include confidence.
- Confidence is not a probability of profit. It is the agent's confidence that the rule conditions are cleanly met.

### `backtest-results.tsv`

Purpose:

- stores fixed, comparable metrics.

Columns:

```text
run_id	sample_id	ts_code	signal_date	rule	action	confidence	return_n1	return_n3	return_n5	match_n1	match_n3	match_n5	status	description
```

Status values:

- `promote`: stable improvement versus baseline and enough sample coverage.
- `observe`: promising but not enough evidence.
- `discard`: weaker than baseline or unstable.
- `invalid`: not enough data, rule not calculable, or sample out of range.

### `failure-cases.jsonl`

Purpose:

- captures false positives, false negatives, and unstable cases for later analysis.

Each line:

```json
{
  "run_id": "run-001",
  "sample_id": "000001.SZ-20240102-N10",
  "rule": "retained_accumulation_buy_watch",
  "action": "BUY",
  "confidence": 0.71,
  "failure_type": "buy_false_positive",
  "return_n1": -0.012,
  "return_n3": -0.021,
  "return_n5": -0.018,
  "suspected_reason": "High-chip accumulation rose faster than retained low chips; price breakout may have been distribution rather than accumulation."
}
```

### `final-report.md`

Purpose:

- human-readable summary of the run.

Required sections:

- scope,
- data quality summary,
- baseline result,
- candidate rule results,
- promoted rules,
- observed rules,
- discarded rules,
- invalid samples,
- failure patterns,
- next experiment ideas.

## Agent Roles

### 1. Source Trace Agent

Responsibility:

- ensures every factor and rule has traceability.

Inputs:

- `docs/references/chip-change-strategy-traceability.md`
- candidate rule definitions
- feature design docs

Outputs:

- evidence-level labels,
- missing-source warnings,
- source references for agent decisions.

Hard rule:

- A rule without traceability can only enter the experiment pool as `PROJECT_HYPOTHESIS`.

### 2. Feature Engineer Agent

Responsibility:

- maps source concepts to deterministic features.

Inputs:

- Tushare chip rows,
- daily price bars,
- `ChipChangeFeatureSet` design.

Outputs:

- feature snapshots,
- quality warnings,
- formula notes.

Hard rule:

- This agent does not output trading actions.

### 3. Hypothesis Agent

Responsibility:

- proposes and evaluates candidate strategy rules from features.

Inputs:

- feature snapshots,
- traceability entries,
- previous run results.

Outputs:

- candidate rule signal,
- confidence,
- concise reasoning summary,
- expected failure risks.

Hard rule:

- This agent cannot promote a strategy. It can only propose or apply candidate rules.

### 4. Backtest Analysis Agent

Responsibility:

- compares candidate signals against `N+1`, `N+3`, and `N+5` outcomes.

Inputs:

- candidate signals,
- forward returns,
- baseline results.

Outputs:

- `backtest-results.tsv`,
- aggregate result tables,
- promote/observe/discard/invalid recommendation.

Hard rule:

- It cannot change observation windows or scoring definitions during a run.

### 5. Failure Analysis Agent

Responsibility:

- explains where candidate rules fail.

Inputs:

- false-positive and false-negative samples,
- feature snapshots,
- agent decisions.

Outputs:

- `failure-cases.jsonl`,
- rule refinement suggestions.

Hard rule:

- It cannot rewrite history; refinements become new candidate rules in a later run.

### 6. Report Agent

Responsibility:

- creates the human-readable run summary.

Inputs:

- run config,
- API logs,
- feature snapshots,
- agent decisions,
- backtest results,
- failure cases.

Outputs:

- `final-report.md`.

Hard rule:

- It must distinguish observed evidence from speculation.

### 7. Governance Agent

Responsibility:

- prevents unsupported rules from becoming default strategy.

Inputs:

- evidence levels,
- backtest result summaries,
- failure analysis,
- project rules.

Outputs:

- promotion decision,
- risk notes,
- blocked-rule list.

Promotion policy:

```text
PROJECT_HYPOTHESIS
  -> experiment only

FACTOR_FAMILY + stable backtest result
  -> candidate strategy pool

DIRECT_REPORT or OPEN_REPRODUCTION + stable backtest result
  -> baseline candidate

Any rule without backtest result
  -> cannot become default strategy
```

## First Implementation Scope

The first implementation should be intentionally small:

1. Manual stock universe.
2. Manual date range.
3. Fixed sample window `N`.
4. Observation windows `[1, 3, 5]`.
5. One baseline strategy.
6. One or two candidate rules.
7. Full audit logs for every API call, feature snapshot, agent decision, and result.

Recommended initial candidate rules:

- overhead digestion hold-to-buy watch,
- retained accumulation buy watch.

## Open Decisions

- Whether confidence should be rule-condition confidence only, or combine data quality and rule strength.
- Whether each stock/date sample should call the AI agent, or whether the AI agent should only analyze aggregated batches after deterministic rule scoring.
- Whether `api-calls.jsonl` should be generated by the backend service directly or by a research-run orchestrator wrapper.
- Whether research run artifacts should stay under `docs/research-runs/` or move to a top-level `research-runs/` directory once files become large.
